§01Scope: the gap between keyboard and last push

Pando is a working-tree continuity fabric for one human (plus their agents) across many machines. Every machine renders the same working universe: dirty files, untracked files, unpushed branches, stashes, index state, secrets, and the recipes for derived state. Git remains untouched on top — Pando lives strictly underneath it.

Everything committed and pushed is explicitly out of scope: that is git's and GitHub's job, and Pando leans on it (§04). Pando's entire storage, sync, and trust surface is the delta between the keyboard and the last push.

Identity

Git is history. Pando is presence. Pando moves state; git moves meaning. Any feature that would make Pando impersonate git (auto-pull, auto-merge of team code, commit rewriting) is out of scope by principle.

Non-goals: multi-user real-time collaboration (Google-Docs-for-code), replacing the PR workflow (Sturdy's fatal mistake), cloud-hosted development environments, general-purpose file sync.

§02Core concepts

TermDefinition
fabricThe whole system: daemons + authority + stores. The root system of the grove.
trunkA machine enrolled in the fabric. Renders the working universe locally; holds a device key.
snapshotAn immutable, content-addressed capture of a repo's full state at a quiescent moment. Taken every few idle seconds. Forms a DAG (growth rings).
overlayThe portion of a snapshot not derivable from GitHub: snapshot = base commit SHA + overlay. The only thing Pando stores durably.
authorityThe always-on holder of latest state + lease table. Default: Pando cloud. Promotable to a user's own machine (§06).
leasePer-repo write token. One trunk writes at a time; acquisition happens implicitly on sit-down. Prevents conflicts instead of resolving them.
forkDeliberate divergence: editing while an offline trunk holds unsynced changes. Recorded as two DAG branches; reconciled by three-way merge.
recipeThe generator of derived state: lockfiles + platform triple + tool version. Synced instead of the artifacts it produces.
artifact CASContent-addressed store of built dependencies, keyed by (content hash, platform). Shared across all users for public packages.
exposureOverlay size ≈ how much work exists only in Pando (unpushed). Surfaced in UI; pushing drains it to ~zero.

§03Architecture decisions

Each decision lists the options weighed. ✓ marks the choice; rationale follows.

D1 · TOPOLOGYHow do bytes move between machines?
✕
Pure P2P mesh (Syncthing-style). Requires two machines awake simultaneously — "close lid, open other laptop" breaks. That failure mode is the product failing.
✓
Hub-and-spoke through an always-on authority. Devices never need to co-exist in time. v1 ships this.
→
Later: + opportunistic P2P (LAN direct transfer, Tailscale-style) as a bandwidth optimization — never load-bearing.
VerdictThe always-on authority is what makes walking away safe. Everything else is optimization.
D2 · DETECTIONHow are changes detected?
✕
Periodic scan only (rsync-style). Lag between edit and capture is where data loss lives.
✓
FS watchers (FSEvents / inotify) + chunked content-addressing, with a periodic full scan as integrity backstop for missed events.
✕
Virtual filesystem (FUSE / File Provider / EdenFS-class). Years of kernel-adjacent pain — and unnecessary: source trees are small enough to replicate fully (see D5 insight).
VerdictWatcher + backstop scan. Dura and Mutagen prove the watcher discipline works in production.
D3 · DATA MODELWhat is "state"?
✕
Per-file versions (Dropbox model). No whole-repo consistency; restores produce Frankenstein trees. Code files only make sense together.
✓
Snapshot DAG: immutable content-addressed whole-tree captures every few idle seconds — a shadow git one level below git. Divergence = branching, a solved mental model.
✕
CRDTs / operation logs. Solves two-humans-one-file, which is not this product. Silent semantic merges of code are worse than honest conflicts.
VerdictSnapshot DAG. Reuses git's proven Merkle-tree storage model; gives time travel over uncommitted work for free.
D4 · CONFLICTSTwo machines changed the same repo while apart — now what?
✕
Last-writer-wins. Silent data loss. Disqualified, full stop.
✕
Conflict copies (main (conflicted copy).ts). Stray files poison builds, imports, tooling.
✓
Leases as the normal path: sit-down acquires the per-repo write lease; previous holder released on idle/lid-close. Conflicts are prevented, matching the physics of one human body.
✓
Fork-and-merge as the escape hatch: editing past an offline lease-holder knowingly forks the DAG. Auto-merge when non-overlapping (the common case); one explicit reconcile UI otherwise.
VerdictPrevention first, merge second, silence never. Agents lease their own branches/worktrees and never contend with the keyboard.
D5 · REPLICATIONFull copies or virtual placeholders?
✓
Full local replicas of source on every pinned machine. Key insight: source minus derived state is tiny (MBs). Local-first survives planes and dead wifi.
✕
Files-on-demand placeholders. Only needed if trees were huge; they aren't, because derived state never syncs. Avoids the D2 virtual-FS trap entirely.
VerdictReplicate source fully; keep bulk (history, artifacts) in the CAS. Per-repo pinning per machine.
D6 · GIT RELATIONSHIPWhat does Pando do with .git itself?
✓
Sync .git as ordinary data — safe only under D4's lease. Branches, stash, index, remote-tracking refs travel with you. (Syncthing's documented repo-corruption failure is exactly what the lease removes.)
✕
Replace or wrap git (jj-style). Wrong fight: users' git workflow is sacred (§01 identity).
VerdictSync it under the lease; be git-aware only for divergence merges and the §04 policies. D4 and D6 interlock — neither is safe alone.
D7 · IMPLEMENTATIONWhat is the core implementation language?
✕
Go. Faster initial service iteration, but weaker state modeling for a daemon whose primary promise is never losing work.
✓
Rust. One implementation for the simulation harness, daemon, authority, CLI, and TUI; explicit types around leases and snapshot state; native binaries for every trunk.
VerdictRust from Phase 0 onward. Avoid an early prototype rewrite; keep the dependency and async surface small.

§04GitHub as base layer

Assumption: nearly every repo has a remote. Then everything pushed is already replicated and CDN-distributed — so Pando stores only overlays: snapshot = base SHA + overlay.

  • Storage & trust collapse. Pando holds MBs per user, all E2E-encrypted deltas — never the codebase. Hydrating a new machine = git clone from GitHub's CDN + a tiny overlay from the authority.
  • Overlay GC on push. When a push advances the base, the authority garbage-collects everything now reachable from a pushed ref. Pushing drains exposure toward zero — the system nudges safe behavior without forcing it.
  • Unpushed branches are the overlay's reason to exist: commits/refs unreachable from any pushed ref ride along automatically; orphan histories degrade to full snapshots inside the overlay.
  • Remote drift policy: auto-fetch always, auto-pull never. Fetch is non-mutating; continuous background fetch means machines share one view of origin and pulls are instant. Reconciling with the team's code remains a deliberate git action.
  • Force-push rescue (must ship day one). A non-fast-forward ref move can make an overlay's base unreachable → eventually GC'd by GitHub. On detection, promote the endangered base objects into the overlay before they vanish; slim down after the user rebases.
  • Escape hatch (durability story): optional slow-lane push of encrypted snapshot bundles to hidden refs (refs/pando/…) every ~10 min — recent working state recoverable from GitHub alone if Pando's service disappears.
  • Graceful degradation: no-remote repos get full snapshots. GitHub is an optimization, not a dependency; GitLab/Bitbucket/bare remotes work identically.
Decision

Own authority for live sync; GitHub for base layer + escape hatch. GitHub-as-sole-authority was rejected: push latency (seconds, not ms), API rate limits, secrets-in-repo spookiness, read-only upstream repos.

§05Classification & rehydration

Every path in a working tree falls into one of four buckets:

source

Your code. Syncs always, keystroke-fresh, full replicas everywhere.

secrets

.env & friends. Sync always, E2E-encrypted, never touch GitHub.

derived

node_modules, venvs, target/. Never synced as bytes — the recipe syncs; artifacts rebuild per platform.

local-only

Sockets, .DS_Store, editor caches, logs. Ignored, never rehydrated.

Why rehydrate instead of copy

Derived dirs are outputs of a pure-ish function of (lockfile, platform triple, runtime version) — so memoize per platform rather than transport. venvs prove this is architecture, not optimization: they hardcode absolute paths and break even between identical machines.

  • Trigger: repo materializes or lockfile changes → run the recipe (npm ci, uv sync) in the background, ideally before the editor opens.
  • Artifact CAS: pure-JS packages/wheels shared across platforms and users (public bytes aren't secrets); native modules keyed by platform triple. Installs become hardlinks + a few downloads — the pnpm-store trick, machine-spanning.
  • Pre-warm tier (optional): an always-on builder (user's box or cloud) hydrates its platform variant the moment a lockfile changes elsewhere. Floor is hydrate-on-arrival from warm CAS (seconds); pre-warm upgrades that to zero. Repos' own CI caches (GitHub Actions) can serve as builders.

Implementation status: Phase 1 detects root and nested npm/uv projects, invokes npm ci or uv sync --frozen directly without a shell, and caches successful manifest-plus-lockfile fingerprints outside the repository. pando hydrate runs explicitly; pando watch --rehydrate runs changed recipes after remote materialization. Failures remain retryable and derived-tree watcher events are ignored. Execution is deliberately opt-in because package managers may use the network and execute trusted repository hooks. Non-blocking execution, broader recipe coverage, and a per-platform artifact cache remain follow-up work.

Ignore rules

Portable by default, with conservative built-in defaults for known derived and machine-local paths (node_modules, .venv, root target, tool caches, OS clutter), user-wide rules in ~/.config/pando/ignore, and per-repo .pandoignore; both files use Git-ignore syntax. Precedence runs built-in → user-wide → repository. Snapshot manifests carry the flattened policy so receivers classify and preserve local state consistently. Pando deliberately does not inherit .gitignore: Git-ignored secrets and working state such as .env are portable unless explicitly excluded. pando classify reports the winning rule and source without mutating the tree.

§06Topology & roles

Three jobs exist; the fabric fills each with whatever the user has, defaulting to cloud. Roles, not machines.

RoleJobDefault (Tier 0)Promotable to
authorityAlways-on latest state + lease table; two machines never need to be awake togetherPando cloud (ciphertext only)User's always-on box; cloud drops to thin relay
archiveFull snapshot history; laptops stay thinPando cloud object storageBig-disk home machine
builderPre-warm derived state per platformCloud builders + global artifact CASUser hardware / repo CI

Tier 0 = cloud fills everything (the product must be great here — most users own one laptop). Tier 1 = user promotes a machine (pando promote linuxbox --roles=authority,archive,builder). Tier 2 = hybrid, roles filled independently. Pre-warming is always an optimization tier, never a requirement.

§07Security model

Implementation status: Phase 1 wraps every authority RPC in an authenticated Noise NNpsk0 session using a shared 32-byte fabric key and ChaCha20-Poly1305. Wrong-key, tampered, and legacy plaintext traffic is rejected. The two-Mac self-hosted topology treats the authority as trusted, so plaintext authority storage is acceptable here. Ciphertext-only storage, individual enrollment, and revocation become release gates before any hosted or otherwise untrusted authority.

  • E2E by default; the authority stores ciphertext. Not a feature bullet — qualification to exist, and the moat platforms won't cross (they want to read working trees for AI context; Pando cryptographically can't).
  • Keys: per-user master key → deterministic per-chunk keys (preserves within-account dedup). Device enrollment via existing-device approval (Tailscale/iMessage pattern). Compress before encrypt.
  • Recovery: recovery phrase + N-device redundancy. Lose everything → overlays are gone, but blast radius is unpushed work only; the pushed base survives in plaintext on GitHub. Recovery UX gets first-class design attention — it's where E2E products die.
  • Plaintext-by-design exceptions: the public artifact CAS (public npm/PyPI bytes; global dedup is the win) and control-plane metadata (machine names, lease timestamps, opaque chunk hashes — never content).
  • Perf note: AES-GCM ≈ GB/s per core; payloads are KBs. Crypto ≈ microseconds against 20–100ms network RTTs. E2E's real costs are feature constraints (no server-side compute on content) and recovery UX — not milliseconds.

Per-user cost: p50 ≈ 100–500 MB steady-state (≈ a penny/month at object-storage prices); p99 ≈ 5–20 GB (~$0.30/month) driven by force-push rescues, no-remote repos, media-in-git. Retention: seconds-granularity 7 days → hourly 30 days → daily beyond; quotas + nudges for pathological repos. Compare Dropbox's TBs-per-user against $10/month: the overlay model's unit economics are structurally better — a direct dividend of §04.

§11Risks

Platform risk

GitHub/Microsoft can enter uncommitted territory any quarter (Codespaces already holds dirty trees; VS Code ships to everyone). The moat is what they won't do: E2E they can't read (starves Copilot context), vendor neutrality (GitLab/Bitbucket), local-first against their cloud gravity, and unglamorous cross-platform daemon grind. Lean on all four in design and positioning.

Monetization risk

Individual devs feel this pain hardest and pay worst — exactly why CDE vendors fled to enterprise agent-infra. Likely commercial wedge: working-tree continuity for agent fleets. Personal-tool value is the hedge: worst case, infrastructure I use daily.

Trust risk

A sync tool gets one chance: eat work once, lose the user forever. Mitigations: snapshots are append-only (nothing destructive by construction), simulation harness before real repos, dogfood soak time on the critical path, escape hatch independent of the service.

§12Roadmap & task tracking

Completed project tasks are checked in this specification; additional personal progress persists in this browser (localStorage). Each phase ends in a gate — an explicit decision, not a vibe.

Phase 0 — Prove the magic

1–2 weekends · two machines, my repos, no encryption · deliverable is the feeling: stop editing on one machine, open the other, it's all there.
Gate passedThe live two-Mac handoff worked: dirty files, branch, index, and stash arrived on the second machine. Longer dogfooding and the remote-dev control experiment move to Phase 1 because Phase 0's plaintext transport and full-tree replication were intentionally not daily-driver safe.

Phase 1 — Daily driver I trust

4–8 weeks part-time, dogfooding throughout · calendar dominated by encountering edge cases, not fixing them.
GateThe retention test: after a month, can I tolerate going back to plain git-push discipline or remote-dev? Physically refusing to churn = strongest possible signal. Only then does Phase 2 exist.

Phase 2 — Something a stranger installs

~3–6 months from start · enter only if the P1 gate passed.
GateFirst 10 external users retained for a month without hand-holding → decide: product push (agent-fleet wedge) vs. excellent personal/OSS tool.

§13Open questions

  • Lease UX details: what exactly is "sitting down" (first keystroke? editor focus? fs write?) and "walking away" (idle minutes? lid event?) — needs P0 dogfooding to answer honestly.
  • Session context scope: do open editor buffers / terminal history / running dev-server state belong in v1, or is that a later "session follows you" layer above the file fabric?
  • Local dev databases: seeded Postgres/SQLite state is technically "source-ish local state" — sync (big), snapshot on schedule, or recipe-ify (seeds/fixtures)?
  • Naming collisions: pando.com is taken (career SaaS); trademark search in Classes 9/42 before any commercial move.
